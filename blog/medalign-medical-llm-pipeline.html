MedAlign: Training a Medical AI on a Laptop — From TinyLlama to Production QA System

Transform a 1.1B parameter model into a clinically-aware QA system using QLoRA, DPO alignment, hybrid RAG, and GGUF quantization — all on a 4GB GPU.

The Motivation

Medical AI is dominated by GPT-4-class models. Every query costs $0.03 per 1K tokens. Every patient question travels through an external API. Every piece of sensitive health data leaves the premises, raising serious HIPAA compliance concerns. For clinics in rural areas, developing nations, or any environment without reliable internet, these API-dependent systems simply do not work.

What if you could train a medical QA model that runs entirely offline, on consumer hardware, at zero cost per query? What if a 1.1B parameter model — small enough to fit on a laptop — could answer medical questions with clinically useful accuracy? That is not a hypothetical. That is what MedAlign delivers.

This blog documents the complete pipeline: from curating 60K medical QA pairs, through QLoRA supervised fine-tuning and DPO preference alignment, to hybrid RAG with cross-encoder reranking, GGUF quantization for edge deployment, and rigorous evaluation on three medical benchmarks. Every component runs on a 4GB VRAM GPU. The entire system is open source.

The Challenge

The Challenge is building a medical QA system from a generic 1.1B model requires solving five interconnected problems simultaneously. Solving any single problem in isolation produces a system that fails in production.

  1. Knowledge injection — SFT alone produces confident wrong answers. The model learns medical vocabulary and response format but makes dangerous errors: recommending ibuprofen to a patient on warfarin without mentioning the deadly bleeding risk.
  2. Safety alignment — Without DPO, the model exhibits four categories of failure: accuracy without safety, hallucinated confidence, missing referrals, and harmful completeness. Each category can cause patient harm.
  3. Knowledge augmentation — Without RAG, the model cannot access current medical literature. It is limited to whatever was in TinyLlama's 2023 pre-training corpus.
  4. Deployment efficiency — The 2.2GB F16 model cannot serve real-time inference on edge devices. GGUF quantization to Q4_K_M reduces this to 0.7GB while preserving clinical accuracy.
  5. Rigorous evaluation — Without systematic benchmarks, you are shipping on vibes. MedAlign evaluates on three medical benchmarks at every pipeline stage.

Compounding everything: the 4GB VRAM constraint. Standard fine-tuning of TinyLlama-1.1B requires 11-13GB of VRAM. Our GPU has 4GB. That is a 7-9GB deficit that QLoRA must solve.

Lucifying the Tech Terms

Before diving into the blueprint, here are the ten key technical terms you need to understand. Each follows the Definition / Simple Example / Analogy pattern.

1. QLoRA (Quantized Low-Rank Adaptation)

Definition: Fine-tuning a frozen 4-bit quantized model by training tiny adapter matrices that represent only 0.6% of total parameters.

Example: TinyLlama-1.1B has 1.1B parameters. QLoRA freezes them all at 4-bit precision, then adds 6.8M trainable LoRA parameters. Standard SFT needs 11-13GB VRAM; QLoRA needs 3-4GB.

Analogy: Adding sticky notes to a compressed encyclopedia — the book stays read-only, your notes personalize it for medical questions.

2. NF4 (Normal Float 4-bit)

Definition: A 4-bit quantization format optimized for neural network weight distributions. Places its 16 quantization levels at the quantiles of a standard normal distribution.

Example: Weight value 0.2134 maps to nearest NF4 level 0.2461, reconstructed as 0.2141. That is 0.33% error — nearly imperceptible to model quality.

Analogy: Like JPEG compression optimized for photos — more bits where visual detail matters (edges, faces), fewer bits where it does not (uniform sky).

3. DPO (Direct Preference Optimization)

Definition: Teaching a model to prefer good answers over bad ones without training a separate reward model. Uses a closed-form loss function derived from the RLHF objective.

Example: "Continue medication as prescribed" (chosen) vs "Try reducing the dose yourself" (rejected) — learned from 5-10K GPT-4-scored preference pairs.

Analogy: A food critic (GPT-4) taste-tests multiple versions of each dish, scores them on accuracy, safety, groundedness, and completeness. The best and worst become training pairs.

4. GGUF (GPT-Generated Unified Format)

Definition: A compact model format for CPU/edge inference via llama.cpp. Supports multiple quantization levels for size-quality tradeoffs.

Example: TinyLlama F16 (2.2GB) converts to Q4_K_M (0.7GB) — 3x smaller, runs inference on a laptop without a GPU.

Analogy: Converting a Blu-ray movie to a compressed streaming format — smaller file, nearly the same visual quality for practical viewing.

5. Hybrid Retrieval (Dense + Sparse)

Definition: Combining semantic vector search (FAISS dense, weight=0.7) with keyword matching (BM25 sparse, weight=0.3) for better recall than either method alone.

Example: "heart attack treatment" finds both exact keyword matches AND semantically related documents about "myocardial infarction management protocols."

Analogy: Google Search uses both page content matching AND understanding what you meant — combining two complementary signals for better results.

6. Cross-Encoder Reranker

Definition: A model that scores query-document pairs TOGETHER (jointly encoded) for precise relevance scoring, unlike bi-encoders that encode them separately.

Example: Top-50 retrieved documents are reranked to top-10 using ms-marco-MiniLM-L-12-v2. The reranker catches relevance nuances that initial retrieval misses.

Analogy: First-pass resume screening (bi-encoder) vs detailed in-person interview (cross-encoder) in job hiring — different precision levels for different stages.

7. Reciprocal Rank Fusion (RRF)

Definition: A method to merge ranked lists from multiple retrievers using inverse-rank weighting with constant k=60.

Example: FAISS ranks doc #42 at position 3, BM25 ranks it at position 7. Fused score = 0.7/(60+3) + 0.3/(60+7) = 0.0111 + 0.0045 = 0.0156.

Analogy: Combining expert panel scores — each expert's ranking contributes proportionally, with higher-ranked items getting exponentially more weight.

8. ChatML Format

Definition: Structured prompt template with <|im_start|>system/user/assistant<|im_end|> role markers that define the conversation structure.

Example: System="You are MedAlign, a helpful and accurate medical assistant..." User="What is diabetes?" Assistant="Diabetes mellitus is a chronic metabolic condition..."

Analogy: Like a play script with stage directions — each actor knows their role, their cues, and when to speak.

9. PubMedQA

Definition: A biomedical QA benchmark with 500 labeled yes/no/maybe questions derived from PubMed research abstracts.

Example: "Does vitamin D supplementation reduce cardiovascular risk?" Given the abstract context, the model must answer "yes," "no," or "maybe" based on evidence.

Analogy: A medical board exam where the answer must be supported by the provided research paper — no guessing, no outside knowledge, just evidence-based reasoning.

10. MinHash Deduplication

Definition: A probabilistic technique to find near-duplicate documents using hash-based fingerprints in O(n) time, far faster than O(n²) brute-force comparison.

Example: 60K medical QA pairs contain 14,800 noise samples (24.7%): 8,000 exact duplicates, 4,000 near-duplicates, 1,500 with HTML junk, 800 too short, 300 too long.

Analogy: Library fingerprinting — instead of reading every book cover to cover, sample 128 random pages from each to create a compact signature for comparison.

Making the Blueprint

MedAlign's pipeline has eight stages. Each stage builds on the previous one. Here is the complete blueprint from raw data to deployed system.

Step 1: Data Curation & Preprocessing (60K to ~50K Clean)

MedAlign combines 60,000 medical QA pairs from three HuggingFace datasets, each covering a different aspect of medical knowledge:

  • Medical Flashcards (30,000) — medalpaca/medical_meadow_flashcards — Clinical knowledge Q&A with structured question-answer format
  • WikiDoc (10,000) — medalpaca/medical_meadow_wikidoc — Medical encyclopedia entries covering broad medical terminology
  • ChatDoctor (20,000) — lavita/ChatDoctor-HealthCareMagic — Patient-doctor conversations capturing real-world interaction patterns

Each entry has two columns: instruction (the medical question) and output (the expected response). But 24.7% of the raw data is noise.

The Preprocessing Pipeline (4 operations in sequence)

  1. Unicode Normalization (NFKC) — "fibrillation" with ligature characters becomes "fibrillation." Ensures consistent tokenization.
  2. HTML Tag Removal — <b>severe</b> becomes "severe." Prevents the model from generating HTML in responses.
  3. URL Removal — Removes PubMed links that would cause hallucination when the model tries to generate URLs it memorized.
  4. Whitespace Normalization — Collapses multiple spaces and normalizes line breaks for consistent tokenization.

MinHash Deduplication

The three datasets were created independently but draw from similar medical knowledge. The same question about type 2 diabetes treatment appears in three different phrasings. MinHash finds these near-duplicates in O(n) time using 128 random hash functions and Jaccard similarity thresholds:

  • 8,000 exact duplicates — identical question-answer pairs across datasets
  • 4,000 near-duplicates — rephrased versions of the same medical knowledge
  • 1,500 entries with HTML junk — residual markup from web scraping
  • 800 entries too short — one-word answers like "Ok" or "Yes"
  • 300 entries too long — entire textbook chapters pasted as answers
  • Total noise: 14,800 samples = 24.7% of raw data

Step 2: QLoRA Supervised Fine-Tuning

Standard fine-tuning of TinyLlama-1.1B requires 11-13GB of VRAM. Our GPU has 4GB. QLoRA solves this with three innovations stacked together:

  1. LoRA (Low-Rank Adaptation) — Instead of updating all 1.1B parameters, add tiny trainable adapter matrices. Result: 1,100,000,000 params down to 6,800,000 trainable (0.6%). Like modifying 58 words out of every 10,000.
  2. NF4 4-bit Quantization — Store the frozen model in 4-bit Normal Float format instead of 16-bit. Result: 2.2GB model weights down to 0.7GB. NF4 places its 16 quantization levels at quantiles of the standard normal distribution, packing more precision near zero where 68% of neural network weights live.
  3. Double Quantization — Quantize the quantization constants themselves. Saves an additional ~0.1GB. A second-order optimization that squeezes out the last bit of memory.

LoRA Configuration

From configs/train/sft.yaml:

model:
  base_model: TinyLlama/TinyLlama-1.1B-Chat-v1.0
  lora:
    r: 16              # Rank of decomposition
    lora_alpha: 32     # Scaling factor (alpha/r = 2.0)
    target_modules:    # All 7 linear layers per transformer block
      - q_proj         # Query projection (attention)
      - k_proj         # Key projection (attention)
      - v_proj         # Value projection (attention)
      - o_proj         # Output projection (attention)
      - gate_proj      # Gate projection (MLP)
      - up_proj        # Up projection (MLP)
      - down_proj      # Down projection (MLP)
    lora_dropout: 0.05
    bias: none
    task_type: CAUSAL_LM
  quantization:
    load_in_4bit: true
    bnb_4bit_quant_type: nf4
    bnb_4bit_compute_dtype: bfloat16

With rank r=16 and 7 target modules across TinyLlama's 22 transformer layers, that is 154 LoRA adapter pairs (A and B matrices). LoRA B is initialized to zeros — so at the start of training, the model behaves exactly like the original TinyLlama. The adapters gradually learn medical specialization during training.

Step 3: DPO Preference Alignment

SFT teaches WHAT to say (knowledge + format). DPO teaches what NOT to say (judgment + safety). This distinction is critical in medical AI where a technically accurate answer can kill a patient.

The Four Failure Categories of Unaligned Medical AI

Category 1: Accuracy Without Safety

Question: "What medications interact with metformin?"

BAD (accurate but harmful):
"Alcohol can interact with metformin and cause lactic acidosis. However, moderate drinking is generally considered acceptable."
GOOD (accurate AND safe):
"Several medications interact with metformin: Alcohol can cause dangerous lactic acidosis — your doctor should advise on safe limits based on your kidney function. Contrast dyes require pausing metformin before CT scans. Always inform your healthcare provider of ALL medications."

Category 2: Hallucinated Confidence

Question: "Is hydroxychloroquine effective for COVID-19?"

BAD (confidently wrong):
"Yes, hydroxychloroquine has shown significant effectiveness against COVID-19 in multiple clinical trials."
GOOD (honest uncertainty):
"Multiple large randomized controlled trials (RECOVERY, WHO Solidarity) found no clinical benefit of hydroxychloroquine for COVID-19 treatment. Major medical organizations do not recommend its use for this purpose."

Category 3: Missing Referrals

Question: "I have sudden severe headache, stiff neck, and fever"

BAD (treats as routine):
"These symptoms could indicate a viral infection. Rest, stay hydrated, and take acetaminophen for fever."
GOOD (recognizes emergency):
"These symptoms together (sudden severe headache + stiff neck + fever) can indicate MENINGITIS, which is a medical emergency. Please seek immediate emergency medical care (call 911 or go to the nearest ER). Do not wait."

Category 4: Harmful Completeness

Question: "What is the lethal dose of acetaminophen?"

BAD (too complete):
"The lethal dose of acetaminophen is approximately 150mg/kg in adults. For a 70kg person, this would be about 10.5 grams..."
GOOD (responsible):
"Acetaminophen overdose is a serious medical emergency that can cause fatal liver failure. If you or someone you know has taken too much, call Poison Control (1-800-222-1222) or 911 immediately. The safe maximum daily dose is 3,000mg for most adults."

RLHF vs DPO: Why DPO Wins for MedAlign

RLHF (Old Way)

  • 4 stages: SFT → Reward Model → PPO → Eval
  • 3 models in memory simultaneously
  • ~10.4GB VRAM (impossible on 4GB)
  • Unstable PPO training (reward hacking)
  • 10+ sensitive hyperparameters
  • Weeks of training, expensive reward model

DPO (MedAlign)

  • 2 stages: SFT → DPO Training
  • 2 models (policy + frozen reference)
  • ~4GB VRAM (fits our GPU!)
  • Stable supervised learning (no RL)
  • 1 key hyperparameter (beta=0.1)
  • Days of training, no reward model needed

The breakthrough: Rafailov et al. (2023) proved that the partition function Z(x) cancels out in the DPO derivation, enabling a closed-form solution. The entire RLHF objective reduces to 6 lines of code:

# DPO Loss Function — 6 lines that replace the entire RLHF pipeline
def dpo_loss(policy_chosen_logps, policy_rejected_logps,
             reference_chosen_logps, reference_rejected_logps, beta=0.1):
    chosen_rewards = beta * (policy_chosen_logps - reference_chosen_logps)
    rejected_rewards = beta * (policy_rejected_logps - reference_rejected_logps)
    loss = -functional.logsigmoid(chosen_rewards - rejected_rewards).mean()
    return loss, chosen_rewards.detach(), rejected_rewards.detach()

Numerical Walkthrough

For a blood pressure medication question with beta=0.1:

DPO Loss Calculation
chosen_rewards  = 0.1 x (-45.2 - (-47.1)) = 0.1 x 1.9 = 0.19
rejected_rewards = 0.1 x (-52.8 - (-50.3)) = 0.1 x (-2.5) = -0.25
reward_margin    = 0.19 - (-0.25) = 0.44
loss             = -log(sigmoid(0.44)) = -log(0.608) = 0.496

Interpretation:
  chosen_rewards > 0:   Policy prefers chosen MORE than reference does
  rejected_rewards < 0: Policy avoids rejected MORE than reference does
  Both signals correct — training is working!

def _reciprocal_rank_fusion(self, dense_results, sparse_results, k=60):
    fused = {}
    dense_rank_map = {idx: rank+1 for rank, (idx, _) in
                      enumerate(sorted(dense_results.items(),
                      key=lambda x: x[1], reverse=True))}
    sparse_rank_map = {idx: rank+1 for rank, (idx, _) in
                       enumerate(sorted(sparse_results.items(),
                       key=lambda x: x[1], reverse=True))}
    for idx in set(dense_results) | set(sparse_results):
        dense_rrf = self.dense_weight / (k + dense_rank_map.get(idx, len(dense_results)+1))
        sparse_rrf = self.sparse_weight / (k + sparse_rank_map.get(idx, len(sparse_results)+1))
        fused[idx] = dense_rrf + sparse_rrf
    return fused

6A: PubMedQA Evaluation Framework

PubMedQA tests whether a model can reason about biomedical evidence. Given a research question and a PubMed abstract as context, the model must classify the answer as "yes," "no," or "maybe." The benchmark has 500 labeled questions, each requiring evidence-based judgment rather than memorized facts.

Evaluation Methodology

From src/evaluation/benchmarks/pubmedqa.py:

def format_pubmedqa_prompt(question: str, context: str) -> str:
    return (
        f"Based on the following medical research context, answer the question "
        f"with 'yes', 'no', or 'maybe'.\n\n"
        f"Context: {context}\n\n"
        f"Question: {question}\n\n"
        f"Answer:"
    )

def extract_pubmedqa_answer(generated_text: str) -> str:
    text_lower = generated_text.strip().lower()
    for label in ["yes", "no", "maybe"]:
        if text_lower.startswith(label):
            return label
        if f"the answer is {label}" in text_lower:
            return label
        if f"answer: {label}" in text_lower:
            return label
    for label in ["yes", "no", "maybe"]:
        if label in text_lower:
            return label
    return "maybe"  # Default fallback

Generation uses greedy decoding (do_sample=False), max 64 new tokens, and truncation to 2048 tokens. Metrics: exact-match accuracy and macro-F1 (average F1 across all 3 classes).

Cross-Benchmark Analysis

Each benchmark tests a different capability. PubMedQA tests comprehension of research abstracts — RAG helps most here because it provides actual abstracts. MedQA USMLE tests clinical reasoning with USMLE-style questions — SFT drives the largest improvement because it teaches structured clinical thinking. MedMCQA tests breadth of medical knowledge across subspecialties.

The consistent pattern: SFT provides the largest absolute gain (knowledge injection), DPO provides calibration and safety (most visible in "maybe" class accuracy and safety-related questions), and RAG provides the most benefit on questions requiring specific literature references.

Monitoring Callbacks

# From src/training/callbacks.py
class MemoryMonitorCallback(TrainerCallback):
    """Monitors GPU VRAM usage during training. Critical for 4GB GPUs."""

    def __init__(self, warn_threshold_gb: float = 3.5):
        self.warn_threshold_gb = warn_threshold_gb

    def on_step_end(self, args, state, control, **kwargs):
        if not torch.cuda.is_available():
            return
        allocated_gb = torch.cuda.memory_allocated(0) / (1024**3)
        if allocated_gb > self.warn_threshold_gb:
            logger.warning(
                f"[Step {state.global_step}] VRAM usage: {allocated_gb:.2f}GB "
                f"(threshold: {self.warn_threshold_gb}GB)"
            )

The MemoryMonitorCallback is essential for 4GB GPU training. It logs a warning whenever VRAM usage exceeds 3.5GB — the 0.5GB buffer prevents OOM crashes that would lose training progress. Combined with the LoggingCallback that tracks loss at each step, these callbacks provide real-time visibility into training health.
